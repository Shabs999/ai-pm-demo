Ingestion history — did every meeting land?

Every fact on the dashboard cites the source meetings it came from. The first citation is the meeting that introduced it; later citations are meetings that touched it again. This section turns that around: one row per ingested artifact, showing what each meeting actually contributed — how many stories, requirements, actions, decisions, risks and so on it introduced, plus how many existing items it deepened (added detail or a fresh source to).

Each “introduced” chip reads stories +1 (11 → 12): the label and how many that meeting added, then the running total before → after this meeting. So you can verify the arithmetic at a glance — “we had 11 stories, this meeting added 1, now 12.” The counts build up oldest meeting first, so the newest row’s “after” values are the current totals: they match the number badges on the dashboard’s own section headers exactly (e.g. the newest milestones “after” equals the count beside Milestones). “deepened” chips carry no totals — they don’t change any count.

Each row is dated by the artifact’s own date — the DATE line from the message header you sent the digester, not the day you downloaded the file. The best fix for a wrong date is at the source: correct the DATE line before you send the message so it lands right the first time. To repair one that already landed wrong, send a small pasted delta that sets just that artifact’s date; the history re-sorts itself the moment the date is right.

A flat user-story count is normal: once the core stories exist, later meetings usually deepen them — adding rules, examples or scenarios — instead of adding brand-new stories. Seeing “deepened” climb while “introduced” stays low is the system working, not a gap.

What is worth a second look: a requirements-heavy meeting whose row shows nothing introduced and nothing deepened. That usually means its transcript was never fully digested. Re-check it with “Copy — check the last digest” from the Copilot menu — it works in any chat. If the original chat is gone, start a fresh chat and attach the transcript file; the check matches it against the digest by file and date, so a meeting can be audited long after its chat was deleted.

Where the status pills come from

Every status pill — on user stories, scenarios and requirements — is extracted from your digested sources, never invented. Items begin life as draft. They are promoted along the pipeline — in review → agreed / approved (and for scenarios, implemented → passing) — only when a source actually records that change. Nothing auto-advances. To see the provenance behind any pill, click the item: its source artifacts are listed, each tracing back to the Artifact feed.

Traceability (RTM)

The traceability matrix — which scenarios cover which story and rule, and where gaps are. Derived automatically from the user stories; it is never maintained by hand.

UAT script

User-acceptance test scripts derived from the Gherkin scenarios. Each case tells the tester why the test exists (story and rule), the preconditions, the steps, and exactly what to look for. Each case traces back through scenario → example → rule → story.
